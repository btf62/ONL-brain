# Agents and Serve Teams

This document keeps the language clear as this repository grows to support both software agents and human ministry contributors.

## Core distinction

- Agents are software workers that review, classify, draft, check, or automate defined tasks in the repo.
- Serve teams are people serving in ministry roles with pastoral, operational, or technical responsibilities.

These two groups may support the same ministry system, but they are not interchangeable.

## Language standard

Use:

- `serve team` and `serve team member` for humans
- `agent` for software processes or AI task workers

Avoid:

- using `volunteer` when the ministry standard is `serve team`
- describing a human role as an agent
- describing an agent as if it carries pastoral authority or relational responsibility

## Typical responsibilities

Agents may help with:

- promotion review between repo and Confluence
- documentation audits
- checklist or template generation
- change detection
- dashboard preparation
- source-system digesting
- online response queue review
- baptism pipeline tracking
- serve team roster reconciliation
- budget watch summaries

Serve team members may help with:

- chat hosting
- moderation
- prayer response
- technical support
- follow-up support

## Responsibility boundary

Agents can support ministry operations.
Serve team members and staff carry ministry responsibility.

That means agents may recommend, summarize, flag, and prepare.
Humans should own discernment, care, leadership, and final ministry decisions.

Personal-support agents may also read repo context to help with clarity, prioritization, emotional regulation, and scope reduction.
Even then, they are still support systems rather than therapists, pastors, or final decision-makers.

## Why this matters

If the language gets fuzzy, it becomes harder to:

- define ownership clearly
- recruit and train people well
- set healthy expectations for automation
- preserve pastoral responsibility where it belongs

## Practical implication for this repo

When documenting a role, decide first whether it is:

- a human role
- an agent role
- a workflow shared by both

If needed later, the repo can grow separate sections for agent cataloging and serve team development.
